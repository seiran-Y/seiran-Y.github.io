Git 分支命名与提交规范

分类： 编程基础
标签： Git代码规范团队协作

日期：2026-06-03 21:00:00

Git 规范不是为了把流程变复杂，而是为了让团队在协作时少猜一点。一个分支名能看出它在做什么，一个提交信息能说明这次修改的目的，一个稳定分支不被随手改动，后面排查问题、回滚版本、发布上线都会轻松很多。

这篇文章结合常见的大厂 Git 代码规范、Git Flow 分支模型、Conventional Commits 提交约定，以及 Git 和 GitHub 官方文档，整理一套适合日常项目使用的分支命名、环境对应和提交信息规范。

项目 Git 整体结构图

普通需求主线：开发、测试、预上线、生产发布
feature/*

功能分支

从 develop 拉出，用来开发单个需求或模块。完成后通过评审合并回 develop，不直接进入生产。
develop

开发集成分支

对应 DEV 环境，汇总最新开发成果。要求基本可运行，作为后续测试和发布的来源。
test

功能测试分支

对应 FAT 环境，用于测试人员做功能测试、联调测试和回归验证。
release/*

预上线分支

对应 UAT 环境，用于上线前验收。此阶段只处理阻塞发布的问题，不再加入大功能。
main/master

生产稳定分支

对应 PRO 环境，保存已发布版本。通常开启分支保护，只接收 release 或 hotfix 合并。
main/master

线上版本来源

线上问题以当前生产代码为基准修复，避免混入未上线功能。
hotfix/*：线上紧急修复分支
从 main/master 拉出，只修复生产环境紧急 bug。修复完成后，一路合并回 main/master 并打补丁版本标签，同时同步回 develop，防止后续版本漏掉这次修复。
develop

同步修复结果

hotfix 回合到 develop 后，后续功能版本也会带上这次线上修复。
开发流向：feature/* -> develop -> test -> release/* -> main/master。
发布流向：release/* 验收通过后合并到 main/master，并打 tag 记录版本。
修复流向：hotfix/* 必须同时回到 main/master 和 develop。

可以把它理解成两条主线：普通需求从 develop 拉出 feature/*，测试通过后依次进入 test、release/* 和 main/master；线上紧急问题从 main/master 拉出 hotfix/*，修复后既要回到生产分支，也要同步回 develop，避免后续版本丢失修复。

一、规范的核心思路

团队里使用 Git，最怕的不是命令不会敲，而是历史看不懂、分支不知道从哪里来、改动不知道要合到哪里去。

比较实用的规范可以围绕三件事展开：

规范对象 解决的问题
分支命名 通过分支名判断代码用途和生命周期
环境对应 明确哪类分支可以部署到哪个环境
提交信息 通过 commit message 看懂每次修改的意图

简单来说，分支负责管理代码流向，环境负责承载验证阶段，提交信息负责记录修改原因。

二、常见分支类型

下面这套分支方式接近 Git Flow。它适合有开发、测试、预上线、生产环境区分的项目。如果项目很小，也可以只保留 main、develop、feature、hotfix 这几类。

1. master / main：生产稳定分支

master 或 main 是主分支，通常对应生产环境代码。

现代很多仓库默认使用 main，早期项目中仍然常见 master。两者职责可以理解为一样：保存已经发布或可以随时发布的稳定版本。

建议规则：

  • 不直接在 master / main 上开发
  • 不直接向主分支提交临时代码
  • 主分支只接收 release 或 hotfix 合并
  • 主分支合并后建议打版本标签，例如 v1.2.0
  • 重要项目应开启分支保护，禁止强推和随意删除

4. test：测试分支

test 分支通常对应功能验收测试环境，也可以叫 fat、qa 或 testing。

它的作用不是继续开发新功能，而是把一批已经完成的功能放到测试环境中验证。测试人员可以在这个环境里做功能测试、回归测试和集成测试。

建议规则：

  • 只合并已经完成开发的代码
  • 不建议在 test 分支上直接写业务代码
  • 测试中发现问题，应回到对应功能分支或 bugfix 分支修复
  • 测试通过后，再进入 release 阶段

三、分支与环境的对应关系

分支和环境要有稳定对应关系。否则同一套代码今天部署到测试环境，明天又被临时改到预发环境，很容易让排查变成猜谜。

常见对应关系如下：

分支 作用 对应环境 是否对外可访问
main / master 主分支，稳定版本 PRO 是
develop 开发集成分支，最新开发版本 DEV 是，通常仅内部
feature/* 功能开发分支，实现新特性 无固定环境 否
test 测试分支，功能测试 FAT 是，通常测试人员访问
release/* 预上线分支，发布候选版本 UAT 是，通常验收人员访问
hotfix/* 紧急修复分支，修复线上问题 无固定环境 否

几个环境名称可以这样理解：

环境 含义
DEV 开发环境，用于开发人员联调
FAT 功能验收测试环境，用于测试人员验证功能
UAT 用户验收测试环境，用于上线前确认
PRO 生产环境，真实用户访问的环境

实际项目里命名可能不同，例如有的团队用 qa 代替 test，用 staging 代替 release。名称可以变，但职责最好不要混。

命名建议：

  • 用 / 做类型分组
  • 用 - 或 _ 连接单词
  • 避免空格、中文、特殊符号
  • 不要使用过长描述
  • 如果有需求编号，放在说明前面或后面都可以，但团队要统一

不推荐：

Git 本身对引用名有格式限制。例如分支名不能包含空格、连续两个点、冒号、问号、星号、反斜杠，也不能以斜杠结尾。为了减少跨平台和工具链问题，日常项目里直接采用“小写英文 + 横线 + 斜杠分组”最稳。

六、常见 commit type

下面这组类型基本覆盖日常开发：

类型 适用场景
feat 新增功能
fix 修复 bug
docs 只修改文档
style 不影响代码含义的格式修改，例如空格、缩进、分号
refactor 既不是修 bug，也不是加功能的代码重构
perf 性能优化
test 添加或修改测试
chore 构建流程、依赖、脚手架、辅助工具等杂项修改
build 构建系统或外部依赖变更
ci CI 配置或流水线变更
revert 回滚之前的提交

feat 和 fix 最常用，也最适合和版本号、变更日志自动生成工具结合。其他类型主要帮助团队快速理解修改范围。

七、单次提交注意事项

一次提交最好只解决一类问题。

推荐原则：

  • 同一次提交的改动必须属于同一类目的
  • 一个提交尽量只对应一个清晰目标
  • 不要把新功能、bug 修复、格式化、重构混在一起
  • 单次提交涉及的问题不要太多，能拆就拆
  • 提交前先看一遍 git diff --staged

十、主分支保护建议

对团队项目来说，规范不只靠口头约定，也要靠工具限制。

建议给 main、develop、release/* 这类关键分支配置保护规则：

  • 禁止直接 push
  • 禁止删除分支
  • 禁止强制推送
  • 合并前必须通过代码评审
  • 合并前必须通过自动化测试
  • 要求分支与目标分支保持最新

这样可以减少“手滑把代码推到生产分支”的风险，也能保证关键分支里的代码至少经过基本检查。
